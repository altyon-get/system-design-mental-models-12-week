Week 01
Scalability Thinking & System Primitives
▼
Why this matters: Every architectural decision — sharding, caching, queue depth — traces back to understanding where your bottlenecks are. Engineers who skip this end up optimizing the wrong layer.
Core Concepts
Horizontal vs vertical scaling · Stateless vs stateful services · The CAP theorem in practice · Latency vs throughput trade-offs · P99/P999 tail latencies · Little's Law (L = λW) · Single points of failure · The fallacies of distributed computing
Modern Production Reality
Kubernetes pod autoscaling · AWS Auto Scaling Groups · Spot instance fleet strategies · Multi-region active-active deployments · Chaos engineering principles (Netflix Chaos Monkey model) · SLOs, SLAs, and error budgets
Technologies to Explore
Kubernetes HPA/VPA AWS EC2 Auto Scaling Load Balancers (ALB/NLB) Terraform (IaC basics) Prometheus metrics Grafana dashboards
Exercises
Capacity estimation drill
Design for "Twitter at 500M DAU" — estimate QPS, storage (tweets, media), bandwidth. Practice back-of-envelope without a calculator for 30 min. Repeat for Dropbox and WhatsApp.
Bottleneck mapping
Take a simple web app (user → API → DB) and draw every possible bottleneck. For each, write the first mitigation you'd apply. Discuss with a peer or write it up.

Week 02
Databases: Choosing, Scaling & Designing Schemas
▼
Why this matters: The database is almost always the hardest part of a system to scale and change. Wrong database choice or schema is debt that compounds for years. In 2026, engineers need to fluently navigate SQL, NoSQL, NewSQL, vector, and time-series stores.
Core Concepts
ACID vs BASE · Read replicas & replication lag · Sharding strategies (range, hash, directory) · Index design (B-tree, LSM-tree) · Connection pooling · MVCC · Write-ahead log (WAL) · Database partitioning · Denormalization trade-offs · Consistent hashing
Database Taxonomy
Relational: PostgreSQL, Aurora · Document: MongoDB, Firestore · Wide-column: Cassandra, DynamoDB · Key-value: Redis, DragonflyDB · Time-series: InfluxDB, TimescaleDB · Vector: Pinecone, pgvector, Weaviate · Graph: Neo4j, Amazon Neptune · NewSQL: CockroachDB, Spanner
Technologies
PostgreSQL + pgvector DynamoDB single-table design Redis Cluster Cassandra ring topology PgBouncer (connection pooling) TimescaleDB
Exercises
Database selection matrix
For each system — Instagram, Uber Surge Pricing, IoT sensor telemetry, a fraud detection model store — pick the right DB and justify it. List trade-offs explicitly.
Schema design under constraints
Design a schema for a multi-tenant SaaS product where each tenant has its own data isolation requirement but shares infrastructure. Model it in both PostgreSQL (row-level security) and DynamoDB (single-table design).

Week 03
Caching Architecture & CDNs
▼
Why this matters: Caching is where you buy 10× performance for a fraction of the cost. But wrong caching strategies (thundering herd, stale data, cache avalanche) introduce subtle production disasters. This week builds the intuition to cache correctly.
Core Concepts
Cache invalidation strategies (TTL, write-through, write-behind, write-around) · Cache-aside (lazy loading) · Read-through · Cache stampede / thundering herd · Hot key problem · Cache warming · Eviction policies (LRU, LFU, ARC) · Multi-level caching (L1/L2/L3 analogy) · Negative caching
CDN & Edge Computing
CDN origin shielding · Cache-Control headers · Surrogate keys (Fastly, Varnish) · Edge Side Includes (ESI) · CDN-based compute (Cloudflare Workers, Lambda@Edge) · Stale-while-revalidate · Cache purging strategies · Regional failover via CDN
Technologies
Redis / DragonflyDB Memcached Cloudflare CDN + Workers Varnish Cache AWS ElastiCache Fastly Surrogate Keys
Exercises
Cache hierarchy design
Design caching for a product page on an e-commerce site: browser cache → CDN → API gateway cache → service cache → DB. Define TTL, invalidation triggers, and stale data tolerance at each layer.
Thundering herd simulation
Sketch how a cache stampede happens when a popular Redis key expires during peak traffic. Then design the mitigation: probabilistic expiration + mutex lock + background refresh. Write pseudo-code.

Week 05
Async Systems: Message Queues, Event Streaming & CQRS
▼
Why this matters: Every system above ~10K RPS has async components. Event-driven architecture decouples services, enables temporal decoupling, and is the backbone of every real-time feature at scale. Kafka alone appears in the architecture of Uber, LinkedIn, Airbnb, and Netflix.
Core Concepts
At-least-once vs at-most-once vs exactly-once delivery · Consumer groups & partition design · Dead letter queues · Backpressure · Outbox pattern (transactional messaging) · Saga pattern for distributed transactions · CQRS (Command Query Responsibility Segregation) · Event sourcing · Change Data Capture (CDC)
Queue vs Streaming Trade-offs
Queues (SQS, RabbitMQ): task distribution, job queues, best for point-to-point · Streams (Kafka, Kinesis, Redpanda): event replay, fan-out, audit logs, time-travel queries · Pub/Sub (Redis Pub/Sub, GCP Pub/Sub): ephemeral fan-out, real-time notifications · Choosing by retention, ordering, replay, throughput needs
Technologies
Apache Kafka Redpanda (Kafka-compatible) AWS SQS + SNS Debezium (CDC) RabbitMQ NATS / NATS JetStream
Exercises
Design an order processing pipeline
Model an e-commerce order flow: payment → inventory → fulfillment → notification. Use the Saga pattern with choreography (event-driven) vs orchestration (central orchestrator). Compare failure modes.
Outbox pattern implementation sketch
Explain how you'd guarantee exactly-once processing of payment events using the transactional outbox pattern. Draw the DB schema, polling worker, and Kafka producer flow.

Week 06
Reliability Engineering: Fault Tolerance, Resilience & Observability
▼
Why this matters: A system that doesn't fail gracefully isn't production-ready. Observability is now a first-class design concern — you can't fix what you can't see. This week directly maps to how senior engineers think about production systems.
Fault Tolerance Patterns
Circuit breaker (Closed/Open/Half-Open) · Bulkhead isolation · Timeout strategy · Retry with jitter · Fallback responses · Graceful degradation · Health check patterns (liveness vs readiness probes) · Blue-green & canary deployments · Feature flags · Dark launches
Observability: The Three Pillars
Metrics: RED (Rate, Error, Duration), USE (Utilization, Saturation, Errors) · Logs: structured logging, log aggregation, correlation IDs · Traces: distributed tracing, sampling strategies, span context propagation · Events: alerting, anomaly detection · OpenTelemetry as the unifying standard in 2026
Technologies
OpenTelemetry Grafana + Prometheus Jaeger / Tempo (tracing) Datadog / Honeycomb PagerDuty alerting Resilience4j / Polly
Exercises
Failure mode analysis
Take your Week 5 order pipeline. For each service, list the top 3 failure modes. Design the circuit breaker thresholds, fallback responses, and alerting rules. What's the minimum graceful degraded state?
Observability instrumentation plan
For a new microservice, define: which metrics to expose (with RED framework), log fields (structured, with correlation ID), and which spans to trace. Write the OpenTelemetry SDK setup in pseudocode.

Microservices & Cloud-Native Architecture
▼
Why this matters: Microservices are the dominant architectural style in 2026 — but also the most commonly botched. Understanding service boundaries (Domain-Driven Design), deployment patterns, and operational complexity is what separates architects from implementers.
Service Design Principles
Domain-Driven Design (Bounded Contexts, Aggregates) · Conway's Law in practice · Service granularity trade-offs · Anti-corruption layer · Strangler Fig migration pattern · Sidecar / Ambassador / Adapter patterns · API composition vs database joins · Data ownership across services · The two-pizza team rule

Week 08
Real-Time Systems: WebSockets, SSE & Live Data Architectures
▼

Transport Protocols

Scaling Real-Time
Fan-out problem (1 event → millions of clients) · Presence systems (who's online) · Collaborative CRDT-based editing (Yjs, Automerge) · Live cursors and conflict-free replicated data types · Connection multiplexing · WebSocket server clustering with Redis Pub/Sub · Fan-out-on-read vs fan-out-on-write trade-off · Backpressure handling

Live collaboration session design
Design the backend for a Google Docs-like collaborative editor for 50 concurrent users per document. Choose between operational transforms and CRDTs. Explain your persistence strategy (how do you save without blocking collaboration?).

Week 12
Design Interview Mastery & Synthesis
▼
Why this matters: Knowing the patterns isn't enough — system design interviews and real design reviews reward structured thinking, explicit trade-off articulation, and the ability to lead a conversation. This week is about building that meta-skill.
The Design Framework
1. Clarify requirements (functional, non-functional, constraints, scale) · 2. Capacity estimation (QPS, storage, bandwidth) · 3. High-level design (diagram with major components) · 4. Deep dive (pick 2–3 critical components) · 5. Trade-offs & bottlenecks · 6. Failure modes & monitoring · Time-boxing each phase
Common Failure Modes in Design Discussions
Over-engineering immediately · Skipping clarifying questions · Not quantifying scale · Designing without considering failure · Ignoring operational concerns (observability, deployment) · Not explaining why you made a choice · Anchoring on one solution too early · Missing the simple solution that actually scales
Week 12 Activities
Mock design sessions × 3
Do 3 full 45-minute mock interviews (with a peer, on Pramp, or with an AI). Use the framework above. Record yourself. Review where you spent time and where you got stuck.
Write-up challenge
Pick any system you've designed during this program. Write a 500-word design doc: problem statement, constraints, chosen approach, alternatives considered, and open questions. This builds the exact skill used in real engineering orgs.
Component deep-dive library
For each major component (Load Balancer, Message Queue, Cache, DB, Search Index), write a 1-page "cheat sheet" from memory. These become your mental scaffolding in interviews.

Hands-On Mini Projects

Build these in the cloud (AWS Free Tier / GCP / Fly.io) to simulate real production constraints. The goal is not perfect code — it's experiencing the operational and architectural challenges firsthand.

4. Search Engine with Hybrid Retrieval Week 9
Build a hybrid search system over a document corpus (e.g., Wikipedia articles): keyword search via Elasticsearch (BM25), semantic search via embeddings stored in pgvector, and a simple re-ranker that blends both scores (RRF). Expose it as an API. Measure recall at k=5 and k=10 for both approaches.

Build a document Q&A system: ingest PDFs (chunking, embedding, indexing into a vector store), retrieve with hybrid search, call an LLM API for generation, stream the response to the client. Add: prompt caching, rate limiting per user, a basic observability dashboard (query latency, cache hit rate, token cost). This is the mini project most relevant to 2026 job descriptions.

Instrument all your mini projects with the full observability stack. Define SLOs for each service (e.g., P99 latency < 200ms, error rate < 0.1%). Set up alerts that would wake you up if they breach. Review your dashboards weekly and find one performance insight from each. This is as close to production SRE experience as you can get without being on-call.

Accelerating Learning with AI & Open-Source Tools

Use these tools to compress experimentation cycles, get instant feedback on designs, and explore systems at a depth that would otherwise take years of production experience.

AI Learning Accelerators
🤖
Claude / ChatGPT — Design Critique Partner
After drafting a system design, paste it into Claude and ask: "What are the top 5 failure modes of this design?", "What would break first at 10× traffic?", or "Argue against this design choice." Use it as a relentless devil's advocate. Ask it to roleplay as a Staff Engineer reviewing your design doc.
⚙️
Claude / Cursor — Rapid Prototyping
Use AI coding assistants to build mini projects 3–5× faster. The goal is the architecture, not the boilerplate. Let AI write the plumbing (DB migrations, API scaffolding) so you can focus on the design decisions (sharding logic, cache invalidation, retry handling).
📝
AI Mock Interviews
Use Claude to run mock system design sessions. Prompt: "You are a Staff Engineer at Google. Interview me on designing a distributed rate limiter. Be rigorous. Ask follow-up questions. At the end, give me structured feedback on my requirements gathering, design choices, and communication." Do this 2–3 times per week in Weeks 10–12.
Cloud Platforms for Experimentation
☁️
AWS Free Tier + LocalStack
Use LocalStack to emulate S3, DynamoDB, SQS, Lambda, ElastiCache locally — zero cost, no provisioning delay. Graduate to real AWS when you need actual scale testing. Focus on: DynamoDB single-table design, SQS + Lambda event processing, ElastiCache (Redis), and API Gateway + Lambda patterns.
🚀
Fly.io / Render — Multi-Region Deployment
Deploy your mini projects on Fly.io to experience multi-region deployments with minimal ops overhead. Fly.io's Anycast routing and WireGuard networking let you genuinely test latency differences between regions — invaluable for understanding geo-distribution trade-offs.
🔬
Grafana Cloud (Free Tier)
Use Grafana Cloud's generous free tier to run a production-grade observability stack (Prometheus, Loki, Tempo, Grafana) without managing infrastructure. This is the fastest way to experience real observability tooling before you need to operate it yourself.
Open-Source Tools to Run Locally
🧪
k6 — Load Testing
k6 is the best tool for understanding how your systems behave under load. Write a 20-line JS script to simulate 1000 concurrent users hitting your service. Watch your P99 latency, error rate, and throughput change. This makes bottlenecks viscerally real. Run load tests against all your mini projects.
🐳
Docker Compose — Local Distributed Systems
Docker Compose lets you run Kafka, Zookeeper, PostgreSQL, Redis, Elasticsearch, and your own services locally in a realistic topology. This is the fastest way to experience distributed system failure modes: kill a container and see what happens downstream.
🔀
Toxiproxy — Fault Injection
Shopify's Toxiproxy lets you inject network faults (latency, bandwidth limits, connection drops) between your services. Use it to test your circuit breakers, retry logic, and timeout handling under realistic failure conditions. This is lightweight chaos engineering you can run on your laptop.
📊
Excalidraw / Mermaid — Design Diagrams
Use Excalidraw for freehand architectural diagrams (fast to sketch, collaborative) and Mermaid for code-as-diagram (sequence diagrams, ER diagrams, flowcharts). Practice drawing your system designs in under 5 minutes — speed matters in interviews and real design reviews.
